# Optimization Model — CP-SAT Formulation

**Status:** v0.1 · Formalises `rulebook.md` for OR-Tools CP-SAT. Target: weekly baseline solve (≈50 workers × 6 days × 2 periods × 2 sessions ≈ 1,200 worker-slots; ≈600–900 tasks) — comfortably small for CP-SAT (<10 s expected).

Two modes share one model:
- **BASELINE**: solve next week from templates + escort table (Friday batch).
- **REPAIR**: re-solve after ChangeEvents with the previous version as reference and minimum-change objective active (see `rescheduling_algorithm.md` for the event-driven wrapper; REPAIR may also run in restricted-neighbourhood form).

---

## 1. Sets and indices

| Symbol | Meaning |
|---|---|
| `W` | workers (active, employment overlaps week), indexed `w` |
| `D` | days of the target week (Mon…Sat), indexed `d` |
| `P = {AM, PM}` | periods, indexed `p` |
| `S = D × P × {1,2}` | session slots, indexed `s`; `slot(d,p,k)` |
| `T` | tasks for the week, indexed `t` — generated by §2 |
| `T_fix ⊂ T` | occurrences of FixedService (E+RO/HC/PC/B/D routes) |
| `T_esc ⊂ T` | escort tasks (one per EscortRequest in week) |
| `T_duty ⊂ T` | centre-duty demand units: one task per (centre, d, p, role, i≤required_count) |
| `T_misc ⊂ T` | kitchen/van/admin template occurrences |
| `C` | centres `{AMC, MRC, GC}` |
| `G` | districts (travel zones) |
| `E` | elders |
| `V0` | previous published version's entries (REPAIR mode) |

Task attributes: `day(t), period(t), sessions(t) ⊆ {1,2}` (or `HALF` for occupies=half_day), `service(t)`, `elder(t)?`, `district(t)?`, `priority(t) ∈ {0..4}` (P0 highest), `pinned_worker(t)?`, `exclusive(t)`, `gender_req(t)`, `time_window(t)?`.

Worker attributes: `gender(w)?`, `avail(w,d,p) ∈ {0,1}` (hours ∧ ¬leave ∧ Saturday team ∧ ¬OFF), `skill(w, svc) ∈ {1,0,unknown}`, `route_ok(w, r)`, `home_team(w)`, `dutycount_hist(w)` (rolling centre-duty count for fairness).

**Fail-safe eligibility** (RB-DATA-01): `elig(w,t) = 1` iff skill confirmed ∧ gender confirmed-or-not-required ∧ route ok ∧ not forbidden by ManualOverride ∧ (¬exclusive(t) ∨ w = pinned_worker(t)). `unknown` ⇒ 0.

## 2. Task generation (pre-model)

1. Expand FixedServices: active-by-effective-date ∧ week_pattern matches this week ∧ no open ServiceCancellationEvent ⇒ one task, `pinned_worker = assigned_worker_id` (soft pin, see §4.9/§5.6), exclusive flag carried.
2. Escort requests in-window ⇒ tasks with `occupies=HALF`, time window `[appt − lead − travel, appt + dur_est]`.
3. Centre duty: `required_count(c,d,p,role)` demand units.
4. Misc template duties (執牌, 跟車, 康雲…) as template occurrences.
5. Apply ManualOverrides: `pin_assignment` ⇒ hard pin; `forbid_assignment` ⇒ remove from `elig`; `cancel` ⇒ drop task, record.

## 3. Decision variables

| Var | Domain | Meaning |
|---|---|---|
| `x[w,t]` | Bool, only for `elig(w,t)=1` | worker w does task t |
| `u[t]` | Bool | task t unassigned |
| `y[w,s]` | Bool | w occupied at session s (channelled from x) |
| `dutyW[w,c]` | Int ≥0 | # duty tasks of w at centre c this week |
| `trav[w,d]` | Int ≥0 | travel penalty of w's day-d chain (table-driven, §5.2) |
| `chg[t]` | Bool (REPAIR) | assignment of t differs from V0 |
| `mv[w,d,g]` | Bool | w visits district g on day d (for travel/ping-pong) |

## 4. Hard constraints

Each cites its rulebook source.

1. **Cover-or-flag** — `∀t: Σ_w x[w,t] + u[t] = 1`. Nothing is silently dropped; `u[t]` is heavily penalised and generates an AuditItem. (RB-DATA-01, infeasibility policy §7.)
2. **Availability** — `x[w,t] = 1 ⇒ avail(w, day(t), period(t)) = 1` (prune vars). (RB-LEAVE-01, RB-TIME-03.)
3. **No double-booking** — `∀w,s: Σ_{t: s ∈ sessions(t)} x[w,t] ≤ 1`; `occupies=HALF` tasks occupy both sessions of the period. When explicit times exist, additionally `NoOverlap` on optional intervals per worker-day with travel buffer `τ(g1,g2)` between consecutive intervals. (RB-TIME-01, RB-TIME-02, RB-ESC-05/06.)
4. **Skill / route / gender / exclusivity** — encoded in `elig` (vars not created). (RB-SKILL-01..04, RB-GEND-01/02, RB-EXCL-01/03.)
5. **Exclusive-cancel** — if `exclusive(t)` and pinned worker unavailable: force `u[t]=1` and tag as `exclusive_cancellation` (do not let another worker take it). (RB-EXCL-02.)
6. **Centre duty floor** — duty demand units are tasks; additionally `Σ_i x[·, duty(c,d,p,role,i)]` shortfall allowed only via `u[t]` (which triggers under-coverage review). Optional NGO-confirmed `min_count` as an absolute floor: `Σ ≥ min_count` (model infeasible below → §7). (RB-DUTY-01.)
7. **Escort quota linkage** — `T_esc` is generated from actual requests, so quota is implicit; extra capacity slots from the template are *not* forced: template `ESC_SLOT` cells only shape `pinned_worker` preferences of escort tasks. (RB-ESC-01..03.)
8. **One escort per worker per period** — follows from HALF occupation. (RB-ESC-08.)
9. **Hard pins** — ManualOverride pins: `x[w*,t] = 1`. Template pins are *soft* by default (see §5.6) except `must`-strength bindings which are hard via `elig`. (RB-CAP-01, RB-EXCL-03.)
10. **Alternate-month exclusion** — task generation already emits only the parity-matching member. (RB-FIX-03.)
11. **Channelling** — `y[w,s] = Σ_{t: s∈sessions(t)} x[w,t]`; `mv[w,d,g] ≥ x[w,t]` for `district(t)=g`; `dutyW[w,c] = Σ_{t∈T_duty(c)} x[w,t]`; `chg[t] = 1 ⇔ assigned worker ≠ V0 worker (or status changed)`.

## 5. Soft constraints → objective terms

Objective: `minimize Σ_k λ_k · Φ_k`. Terms:

| k | Term Φ | Source | Default weight λ (P-tier) |
|---|---|---|---|
| 5.1 unassigned | `Σ_t π(t) · u[t]` with `π` by priority tier: duty 1000, escort 800, exclusive-cancel 0 (forced, reported separately), fixed 500, misc 200 | RB-PRIO-01 | **P0/P1 — dominant** |
| 5.2 travel | `Σ_{w,d} trav[w,d]`; `trav` = Σ pairwise `τ` between consecutive assigned districts (AddElement over the small per-day chains; v1 approximation: Σ over unordered pairs of distinct districts visited + revisit surcharge) | RB-GEO-01/02 | P2: 20 per travel-matrix unit (minutes/10) |
| 5.3 ping-pong | `Σ_{w,d} max(0, Σ_g mv[w,d,g] − 2) · 30` — >2 distinct districts per day surcharge | RB-GEO-02 | P2: 30 |
| 5.4 duty fairness | `Σ_w |dutyW_total[w] + hist(w) − target|` (L1 spread around rolling mean) | RB-DUTY-02 | P3: 10 |
| 5.5 affinity | `Σ_{w,c ≠ home(w)} dutyW[w,c]` | RB-DUTY-03 | P3: 5 |
| 5.6 template stability | `Σ_{t∈T_fix} (1 − x[pinned_worker(t), t])` — deviating from the template worker | RB-FIX-01 | P1-ish: 200 (deviation also triggers review) |
| 5.7 escort preference | `Σ prefer-misses` | RB-ESC-07 | P3: 15 |
| 5.8 workload balance | L1 spread of `Σ_s y[w,s]` across comparable workers | RB-LOAD-01 | P3: 8 |
| 5.9 min-change (REPAIR) | `Σ_t π_c(t) · chg[t]`, `π_c` higher for already-communicated entries (escorts > fixed > duty > misc) | RB-CHG-02 | P2: 100–300 |
| 5.10 refill priority | when releasing ESC slots: bonus for refilling into duty gaps first — emerges from 5.1's π ordering (duty π=1000 > others); no extra term needed | RB-DUTY-04/ESC-02 | — |

Weights are configuration (`solver_config.yaml`), not code; the benchmark suite (`benchmark_cases.json`) locks expected orderings so weight changes are regression-tested. Calibration rule: keep tiers separated by ≥5× so lower tiers can never buy a violation of a higher tier.

## 6. Explainability outputs

Solver must emit, per solve:

1. **Per-entry explanation** (`ScheduleEntry.explanation`): template-follow / refill (which gap, which rule) / displacement (whom it displaced and why allowed) / preference honoured or missed.
2. **Per-unassigned diagnosis** (`u[t]=1`): the *reason set* — computed by re-checking each hard gate for all workers and reporting the binding ones (e.g. "3 candidates skill-qualified; 2 unavailable (leave), 1 gender-mismatch") — this is a cheap post-pass over `elig` and availability, not an unsat core.
3. **Constraint-conflict report** on infeasibility: assumption-based `SufficientAssumptionsForInfeasibility()` over grouped assumption literals (one per constraint family per day) → human-readable "duty floor at GC Wed PM conflicts with 3 leaves" (§7).
4. **Diff report** (REPAIR): entries added/removed/reassigned with triggering event ids.
5. **Metrics block** per `evaluation_metrics.md`.

## 7. Infeasibility handling

CP-SAT with the cover-or-flag pattern (§4.1) is **never globally infeasible** in BASELINE mode: shortage materialises as `u[t]=1` items ranked by π. True infeasibility can only come from hard pins/min_count floors; policy:

1. Solve with all hard constraints. If UNSAT: relax in fixed order (each relaxation = one AuditItem, severity high): (a) drop `min_count` floors → coverage becomes soft; (b) demote `must`-preferences to `prefer`; (c) allow displacement depth 2; (d) allow route-unqualified meal assignment (flagged). Never auto-relax: skill, gender, exclusivity, availability, double-booking.
2. Every relaxation actually used is listed in the version's `metrics.relaxations[]` and requires approval before publication.
3. Time limit 30 s baseline / 5 s repair; on timeout keep best incumbent, mark `metrics.solver_status=FEASIBLE_TIMEOUT`.

## 8. Model-building notes for engineers

- Prune aggressively: create `x[w,t]` only when `elig`∧`avail` — expected density <15%, ~10–20k Bools.
- Use `AddExactlyOne([x[w,t] for w] + [u[t]])` per task; `AddAtMostOne` per (w, s).
- Interval-based NoOverlap only for worker-days containing ≥1 timed task; otherwise session slots suffice.
- Rolling fairness history (`hist(w)`) comes from the last 4 published versions (configurable).
- Seed REPAIR with `AddHint(x[w,t], V0)` for warm starts; typical repair should return in <2 s.
- Determinism: fix `random_seed`, single worker thread for reproducible demos; 8 threads in production solves.
- All weights/π live in config; log the config hash into `ScheduleVersion.metrics` for auditability.
