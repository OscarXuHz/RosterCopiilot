# Evaluation Metrics

**Status:** v0.1 · Computed for every `ScheduleVersion` (stored in `metrics`), asserted by `benchmark_cases.json`, and shown on the review dashboard. All metrics are deterministic functions of (version, demand snapshot, previous version).

## 1. Metric definitions

| Metric | Definition | Direction | Notes |
|---|---|---|---|
| `hard_constraint_violations` | Count of entries violating any P0 rule (skill, gender, exclusivity, double-booking, availability), as found by the shared validator | **= 0 always** | >0 only possible via human override with note; those are listed, not counted as failures |
| `coverage_rate` | assigned demand-backed tasks ÷ all demand-backed tasks (fixed occurrences + escorts + duty units), cancelled-by-event tasks excluded from denominator | ↑, target ≥ 0.98 baseline | |
| `unassigned_count` | `Σ u[t]` split by priority tier `{duty, escort, fixed, misc}` | ↓, target 0 for duty+escort | |
| `center_duty_coverage` | min over (centre, day, period, role) of assigned ÷ required; plus count of slots below required | ↑, target 1.0 / 0 slots | the NGO's top rule |
| `escort_fulfillment_rate` | escort requests with assigned worker ÷ escort requests (net of cancellations) | ↑, target 1.0 | preference-honoured rate reported alongside |
| `total_travel_penalty` | Σ over worker-days of travel-matrix minutes between consecutive assignments (+30/extra district beyond 2) | ↓ | comparable only under same TravelMatrix version — config hash logged |
| `workload_balance_score` | 1 − (σ ÷ μ) of weekly assigned sessions per active FTE-adjusted worker | ↑, target ≥ 0.75 | part-timers scaled by hours |
| `change_distance_from_original` | REPAIR only: # entries whose (worker, slot, service) differs from parent version, weighted by communication cost (escort 3, fixed 2, duty 1, misc 0.5) | ↓ | the "don't churn the published roster" metric |
| `manual_review_count` | blocking items + bulk-approvable items generated by the solve, reported separately | ↓, targets: ≤10 blocking normal week | proxy for admin time actually saved |
| `exclusive_service_cancel_count` | exclusive occurrences auto-cancelled due to worker absence | ↓ (but correctness first: never substituted) | benchmark asserts exact expected counts |

Supporting diagnostics (not targets): `solver_status`, `solve_time_ms`, `relaxations[]`, `data_gap_count` (unknown gender/skill encountered), `import_ambiguity_count`, `template_deviation_count`, `preference_honoured_rate`, `duty_fairness_spread` (max−min rolling duty counts).

## 2. Benchmark protocol

1. Load `mock/canonical.json`, run BASELINE for week 2026-06-08 → assert case `B0`.
2. Apply each EV script (`mock/events/`) on the published baseline → assert its case (`B1`–`B7`).
3. Run the full batch (EV6) both greedy and (Phase 3) CP-SAT REPAIR → both must satisfy the same assertions; CP-SAT must be ≤ greedy on the weighted objective.
4. Importer round-trip: `import(mock/excel/) → canonical'` must equal canonical modulo pathologies; each pathology yields exactly one `import_ambiguity`.
5. Determinism: two runs with the same seed/config produce byte-identical versions.

Assertions use ranges (min/max) rather than exact objective values so weight tuning doesn't break the suite, **except** hard invariants (violations=0, exclusive-cancel counts, blocking-item kinds) which are exact.

## 3. Live-operation health metrics (Phase 4)

Weekly: reviewer decision time per item; % suggestions approved unchanged (target ≥80% — else ranking is wrong); # same-week manual Excel edits outside the system (target →0); data-gap burn-down (unknown genders/skills remaining); roster publication time vs Friday deadline.
